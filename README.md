# 1.Business purpose

## 1) Project background
현대 사회에는 영화, 만화, 드라마 등 수많은 오락거리가 존재하지만, 게임만큼 사용자에게 깊은 몰입감을 선사하고 즐거움, 감동, 성취감 등 다양한 긍정적 감정을 효과적으로 전달하는 매체는 드물다. 특히 코로나 팬데믹 시기를 거치며 온라인을 통한 소통과 각자의 승리나 엔딩을 향해 나아가는 게임의 긍정적인 역할과 수요가 더욱 커졌다. 플레이어들은 정정당당한 경쟁을 펼치고, 그 안에서 승리했을 때 얻는 성취감을 통해 스트레스를 해소하며 활력을 얻게 된다.

국내에서는 이러한 게임 실력을 흔히 '피지컬'이라는 용어로 부른다. 인터넷상에는 반응 속도나 에이밍 등 게임에 필요한 신체 능력을 측정하는 사이트들이 많지만, 대부분 혼자서 결과를 확인하는 수준에 그친다. 전 세계 사람들의 기록을 표로만 제공하다 보니 공정한 측정이 이루어졌는지 신뢰하기 어렵고, 무엇보다 지인들과 실시간으로 비교하며 함께 왁자지껄하게 즐기는 재미를 느끼기에는 한계가 있다.

이러한 아쉬움을 해소하고자, 친구들과 온라인에서 실시간으로 각자의 '피지컬'을 직접 비교하고 경쟁할 수 있는 멀티플레이 게임을 기획했다. 플레이어들은 반응 속도, 클릭 속도, 타자 대결, 에이밍 등 4가지의 직관적인 미니게임에 참여하여 자신의 한계를 시험하고 실력을 겨루게 된다. 또한, '측정'이라는 핵심 테마를 시각적으로 강조하고 특유의 레트로한 감성을 살리기 위해 그래픽은 단순한 흑백 도트 스타일을 채택했다.

## 2) Goal
최대 2명의 플레이어가 하나의 방에 접속해서 실시간으로 데이터를 동기화
플레이어 간의 클릭 속도 및 반응 시간의 오차 범위를 최대한 줄여 공정한 승부 보장
도트 캐릭터의 가이드를 통해 게임의 흐름을 유기적으로 연결하고 경쟁 후 순위표를 제공하여 성취감 극대화 

## 3) Target Market
친구와 가벼운 미니게임을 즐기고 싶은 사람
평소 자신의 게임실력에 자신이 있어서 객관적인 지표를 보여주고 싶은 사람 
평소 도트 그래픽을 좋아하는 사람

# 4. Concept of operation

## 1) 게임 시작화면 진입
| 항목 | 내용 |
| --- | --- |
| Purpose | 사용자가 게임을 실행하고 플레이를 준비 |
| Approach | 게임 실행 후 첫 화면에서 START 버튼을 눌러 게임 시스템에 접근하도록 한다. |
| Dynamics | 플레이어가 게임을 처음 실행했을 때 |
| Goals | 초기 화면을 출력하고 사용자의 입력으로 다음 단계(로비)로 진입한다. |

## 2) 비밀방 생성 및 입장
| 항목 | 내용 |
| --- | --- |
| Purpose | 원하는 인원끼리만 게임을 할 수 있다. |
| Approach | 방장이 비밀번호를 설정하여 방을 만들면 방 목록화면의 방 이름 옆에 자물쇠 아이콘이 표시되며 참가자는 해당 방을 선택하고 올바른 비밀번호를 입력해야만 입장할 수 있다. |
| Dynamics | 로비에서 방을 새로 만들 때, 만들어져있는 비밀 방에 참가할 때 |
| Goals | 플레이어들이 동일한 방에 모여 게임 준비를 마친다. |

## 3) 방 나가기 및 호스트 권한 위임
| 항목 | 내용 |
| --- | --- |
| Purpose | 플레이어가 방을 떠나더라도 남은 인원으로 방이 유지되도록 하기 위함이다. |
| Approach | 플레이어가 방을 떠나 로비로 나갈 수 있고 나간 유저가 방장일 경우 다음으로 입장한 플레이어에게 방장 권한을 자동으로 넘긴다. |
| Dynamics | 방에서 대기 중이거나 게임 씬에서 플레이어가 나가기 버튼을 눌렀을 때 |
| Goals | 방이 사라지지 않고 남은 플레이어들끼리 대기를 할 수 있다. |

## 4) 게임 동시 시작
| 항목 | 내용 |
| --- | --- |
| Purpose | 모든 플레이어가 실제 테스트를 보는것처럼 오차없이 동시에 게임을 시작한다. |
| Approach | 방에 3명이 모두 모이거나 방장이 시작버튼을 누르면 서버를 통해 모든 클라이언트에게 신호를 보내 카운트다운 후 게임을 시작한다. |
| Dynamics | 방에서 게임 시작 조건이 충족되었을 때 |
| Goals | 네트워크 지연을 최소화하여 3명의 화면에서 동시에 게임 시작 및 기록 측정이 시작되도록 동기화한다. |

## 5) 플레이어 입력 처리
| 항목 | 내용 |
| --- | --- |
| Purpose | 플레이어의 플레이 데이터를 정확하게 수집한다. |
| Approach | 마우스와 키보드로 입력되는 사용자의 입력데이터를 시스템이 실시간으로 감지하고 저장한다. |
| Dynamics | 게임이 진행되고 플레이어가 조작할 때 |
| Goals | 플레이어의 입력을 지연없이 정확한 수치로 변환하여 시스템에 기록한다. |

## 6) 실시간 기록 동기화
| 항목 | 내용 |
| --- | --- |
| Purpose | 경쟁의 긴장감을 높이고 다른 플레이어의 점수를 공유한다. |
| Approach | 각 플레이어의 측정 기록을 서버에 전송하여 현재 스코어를 모든 플레이어의 화면에 동일하게 표시한다. |
| Dynamics | 게임 진행 중 점수나 기록에 변화가 발생했을 때 |
| Goals | 모든 플레이어의 화면에 플레이어들의 순위와 기록이 실시간으로 일치하도록 동기화 |

| Dynamics | 모든 게임이 완전히 종료되었을 때 |
| Goals | 최종 순위를 출력하고 우승자에게 성취감을 줄 수 있는 결과 화면을 출력한다. |

# 5. Problem statement
## -Overview
이 게임은 윈도우(PC) 환경 기반이고 포톤 서버를 통하여 최대 2인의 플레이어 간 게임 환경을 실시간으로 동기화한다. 피지컬 측정이라는 게임의 특성상 이 과정에서 여러 가지 문제에 직면할 것으로 예상된다. 
 ● 서버와 플레이어의 동기화 시간 차이
 ● 실시간 동기화 과정에서의 네트워크 과부화 문제
 ● 도트 그래픽의 심미성과 피지컬 게임에서 요구되는 직관적 가독성 간의 충돌 

## 1) Problem 1
반응 속도를 0.001초 단위로 겨루는 미니게임에서 각 플레이어의 인터넷 환경 차이가 승패에 중대한 영향을 미칠 수 있다. 내 화면에서는 내가 먼저 클릭했지만 상대방의 패킷이 서버에 먼저 도착하여 억울하게 패배하는 상황이 발생 할 수 있다. 단순히 신호가 도착한 순서가 아니라 공정한 판정을 내릴 수 있는 기준에 대한 고민이 필요하다.

## 2) Problem 2
타자 대결 미니게임에서 상대방의 타이핑 진행 상황을 내 화면에 실시간으로 띄워야 한다. 이때 플레이어가 키보드를 한 번 누를때마다 서버에 데이터를 전송하면 네트워크 리소스의 심한 낭비와 렉을 유발할 수 있다. 게임의 긴장감은 유지하면서 서버 부하를 최소화 할 수 있는 적절한 방법을 찾아야 한다. 

## 3) Problem 3
레트로한 감성과 측정이라는 분위기를 살리기 위해 단순한 흑백 도트 그래픽을 채택했는데 그래픽이 너무 단순하거나 뭉개지면 플레이어에게 불편함과 피로감을 줄 수 있다. 도트 감성을 유지하면서도 직관적인 가독성을 줄 수 있는 UI/UX가 필요하다.

## NFRs
a. 게임엔진은 유니티를 사용한다.
b. 멀티플레이 네트워크 구축을 위해 포톤 엔진을 사용한다.
c. 반응 속도 측정 오차를 최소화 하기 위해 결과 판정은 포톤 서버의 고유 시간을 기준으로 해서 클라이언트 간의 시간 차이를 보정한다.
d. 네트워크 리소스 최적화를 위해 이벤트 발생 시점에만 데이터를 전송하는 RPC 방식을 사용한다.
